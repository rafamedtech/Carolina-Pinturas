# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

Internal admin panel for Carolina Pinturas (Siigo México dealer). Nuxt 4 + Nuxt UI 4 SSR app. **Siigo is the source of truth for products and customers** (read-only, server-side only); **internal orders live in PostgreSQL/Supabase via Prisma** with full historical snapshots. Siigo México has no orders resource — creating/changing an order only writes to Postgres, never creates invoices or fiscal documents in Siigo.

Spanish is the product language: routes, DB fields, roles, and user-facing strings are Spanish. Match that when adding code.

## Commands

```bash
pnpm dev            # dev server (sets --max-http-header-size for large Supabase auth cookies)
pnpm build          # prisma generate + nuxt build
pnpm lint           # eslint, --max-warnings 0
pnpm lint:fix
pnpm typecheck      # nuxt typecheck (vue-tsc)
```

CI (`.github/workflows/ci.yml`) runs `lint` + `typecheck` only. Full local verification: `pnpm db:validate && pnpm lint && pnpm typecheck && pnpm build`.

### Database

Local Supabase runs on ports **55320–55329** (non-default, to coexist with other Supabase projects). Studio: `http://127.0.0.1:55323`.

```bash
pnpm db:start / db:stop / db:reset
pnpm db:generate    # prisma generate → output at generated/prisma
pnpm db:validate    # prisma validate
pnpm db:drift       # diff datasource vs schema.prisma, fails on drift
pnpm db:lint        # supabase db lint --local
```

**Do NOT use `prisma migrate dev`.** SQL migrations in `supabase/migrations` are the source of truth for the schema; `prisma/schema.prisma` defines the app model. Create migrations with `supabase migration new <name>`, then keep both in sync and verify with `db:validate` + `db:drift`.

## Architecture

### Auth (Supabase Auth, two layers)

- **Server** `server/utils/supabase.ts` → `getVerifiedSupabaseClaims()` verifies the Supabase JWT from cookies (uses `@supabase/ssr` `createServerClient` with `encode: 'tokens-only'`). `server/middleware/supabase-session.ts` runs it on every request to refresh session cookies.
- **App identity** `server/utils/auth.ts`: `getAppSession()` maps the verified `claims.sub` to a row in `public.app_users` (Prisma `AppUser`). Inactive users or unknown roles throw 403. Use `requireUser(event)` / `requireRole(event, roles)` to gate every server route.
- **Client** `app/middleware/auth.global.ts` + `useAuth()` composable guard page navigation. `/lista-precios` is fully public (bypasses both middlewares).
- Public sign-up is disabled; users are created in Supabase dashboard and get an `app_users` row (default role `vendedor`).

### Roles & access control — `app/utils/roleAccess.ts`

Five roles: `admin`, `mostrador`, `vendedor`, `repartidor`, `igualaciones`. This file is the single source of role→permission logic (who can create orders, which order statuses each role may set/edit, per-role home path, per-role path access). **Change permissions here, not scattered across components/routes.** Server routes must independently enforce with `requireRole` — client checks are UX only.

### Siigo integration — `server/utils/siigo.ts`

OAuth token fetched server-side only, cached in module memory (~23h), never exposed to the browser. `siigoConfigured()` gates every Siigo route; when unconfigured the UI shows an explicit connection state instead of mock data. Uses Siigo México contract: `https://api.siigo.mx`, `SiigoAPI-Application-Id` header (default `CarolinaPinturas`).

### Orders — Prisma models in `prisma/schema.prisma`

On save, the server **re-fetches the customer and every product from Siigo** rather than trusting the browser payload, then stores immutable snapshots: `SiigoProduct` keeps queryable columns with prices/taxes/warehouses normalized into related tables; `raw_payload` keeps the full Siigo JSON for forward-compat; each order line snapshots code/name/unit/price/tax/payload as they were at creation. Every status change records timestamp, user, role, and optional note (`SalesOrderStatusHistory`). Order statuses are DB-driven (`order_statuses` table / `OrderStatus` model).

### Layout

- `app/` — Nuxt app: `pages/` (file routes, Spanish paths), `components/` (grouped by feature: `orders/`, `prices/`, `customers/`, `home/`, `inbox/`, `settings/`), `composables/` (`useAuth`, `useOrderDraft`, catalog composables), `utils/` (role/status/auth helpers).
- `server/` — `api/` (h3 route handlers, verb-suffixed files), `middleware/`, `utils/` (prisma client, siigo, auth, validation).
- `generated/prisma/` — generated client (gitignored, regenerated by `db:generate`/`postinstall`).

## Conventions

- **Package manager: pnpm** (`packageManager` pinned). Node 22.
- ESLint stylistic config: no comma-dangle, 1tbs brace style (`nuxt.config.ts`).
- Never expose secrets as `NUXT_PUBLIC_*`: no `sb_secret_`/`service_role` keys, no Postgres URLs. `server/utils/supabase.ts` actively rejects a `sb_secret_` publishable key. Siigo credentials and `DATABASE_URL`/`DIRECT_URL` are server-only.
- `DATABASE_URL` uses the Supabase Transaction Pooler (6543); `DIRECT_URL` uses direct/Session Pooler (5432) for schema commands (prisma.config.ts reads `DIRECT_URL` first).
- Operations that create/modify/void fiscal documents are intentionally disabled until the Siigo contract is validated in a safe test environment.
